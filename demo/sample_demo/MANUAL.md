## 1. このデモは何をするものか

サンプルバックエンド（FastAPI）に対して、**画面から実際に API を叩いて挙動を見せる** ためのモックアップです。用途は主に 3 つ:

1. **プレゼン**：お客様や社内向けに、動く画面を見せながらコンセプトを説明する。
2. **API 動作確認**：エンジニアが開発中の FastAPI エンドポイントを画面から叩いて挙動を確認する。
3. **オフラインデモ**：バックエンドが用意できない環境（新幹線・イベント会場・顧客サイトの持ち込みPC等）でも、**事前に録画したシナリオを再生** することでデモが可能。

依存はゼロ（HTML + CSS + JavaScript のみ）。フォルダをコピーすればどのPCでも動きます。

## 2. 事前準備

- モダンブラウザ（Chrome / Edge / Safari / Firefox の最新版）
- 実バックエンドを叩く場合: Python 3.10+ と サンプル実行環境

| 項目 | 用途 |

| プレゼン用の録画ファイル | 事前に本番と同じシナリオで録画しておく |

## 3. 起動方法（3パターン）

### 方法A: そのままダブルクリック（`file://`）

`demo/sample_demo/index.html` をブラウザで直接開きます。

**できること**: 画面表示、録画の再生、Raw fetch（サーバに繋がる場合）
**できないこと**: 一部のブラウザで実バックエンドへの fetch がブロックされる（CORS/mixed content 制約）

→ **プレゼンで録画再生だけ見せる場合は、この方法で十分**。

### 方法B: 簡易HTTPサーバ経由（推奨）

```bash
cd demo/sample_demo
python3 -m http.server 8000
# ブラウザで http://localhost:8000/
```

**推奨理由**: `file://` 由来の制約がなくなり、実バックエンドとの疎通・録画データ読込がすべて安定します。

### 方法C: NGINX 経由で外部公開

社内 URL や独自ドメインから配信する場合。`SETUP_NGINX.md` の手順に従って、`demo/sample_demo/` を静的配信対象に追加します。

```nginx
location /demo/ {
    alias /path/to/SampleAgent/demo/sample_demo/;
    index index.html;
}
```

**モードバッジ** の色分け:

## 5. ヘッダー操作

### 5-1. Backend URL

デフォルトは `config.js` の `BACKEND_URL`（`http://localhost:8899`）。
テキストボックスを書き換えてフォーカスを外すと、以降その URL に送信されます。**設定はブラウザの localStorage に保存** されるので、リロードしても保持されます（config.js より優先）。

| 用途 | URL 例 |
|------|--------|
| ローカル開発 | `http://localhost:8899` |
| NGINX 経由 | `https://your-domain.com/api` |
| クラウド VM | `https://your-vm.example.com/api` |

### 5-2. Check ボタン + Health ドット

`GET /health` を叩きます。結果によりドットの色が変化:

- ⚪️ グレー = 未確認
- 🟢 緑 = 200 OK
- 🔴 赤 = 到達不可（URL 誤り／サーバ停止／CORS 遮断）

ドットにマウスを合わせるとレスポンス本文またはエラーメッセージが表示されます。

### 5-3. 録画コントロール

| ボタン | 機能 |
|--------|------|
| セレクタ | 再生対象の録画を選択（未選択時は Play 不可） |
| ▶ Play | 選択中の録画を再生 |
| ● Record | 録画開始（タイトルを聞かれる） |
| ■ Stop | 録画または再生を停止 |
| ⬇ Save | 直前に停止した録画を **`.md`（Markdown）** としてダウンロード |
| ⬆ Load | 外部ファイル（`.md` / `.js` / `.json`）から録画を追加ロード |

> **保存形式について**: ⬇ Save は録画を **Markdown（`.md`）** で保存します。ファイル冒頭に人が読める会話トランスクリプトが入り、末尾の ` ```json ` ブロックが再取り込み用のデータ本体です。`.md` は `<script>` で自動読込できないため、再利用は **⬆ Load**（または Recording Editor の Import）から行ってください。`recordings/` に置いて `index.html` の `<script src>` で常設したい場合は、Recording Editor の **⬇ .js** を使ってください。

## 6. 各タブの使い方

### 6-1. Chat (/run) — メインのデモ画面

**やりたいこと**: エージェントとチャットする（=`POST /run`）

**画面レイアウト**:
- 左サイドバー: 実行パラメータ設定
- 中央: チャット履歴と入力欄

**基本操作**:
1. 左サイドバーで **User ID**（例: `DemoUser`）と **Service ID**（例: `DEMO`）を設定
2. **Agent file** ドロップダウンから使うエージェントを選択
3. **Engine (LLM)** ドロップダウンからモデルを選択（未選択の場合はエージェント既定）
4. **Session ID** は空のままにすると新規セッションを自動採番。同じ会話を続ける場合は既存 ID を貼付
5. 実行フラグ（`memory_use` / `rag_query_gene` / `meta_search` / `web_search` / `thinking_mode` / `private_mode`）で挙動を切替
6. 入力欄にメッセージを入力し **Send** ボタン、または `Ctrl+Enter`（Mac: `⌘+Enter`）

**チャット履歴表示**:
- 青い右寄せ吹き出し = ユーザー発話
- グレーの左寄せ吹き出し = エージェント応答
- 各エージェント応答の下に `session=...` メタ情報

**プレゼン向けチップス**:
- **web_search** を ON にしてリアルタイム性を見せる
- **thinking_mode** を ON にして「エージェントが考えている感」を強調
- **private_mode** ON で「メモリに残らない機微な会話」を示す

**添付ファイル（📎 Attach）**:
- 入力欄の上に **📎 Attach** ボタンがあり、複数ファイルを選択できます
- 選んだファイルはチップとして表示され、`×` で個別削除できます
- Send すると:
  - **既定（`Send as base64` OFF）**: ファイルは `multipart/form-data` で **`POST /run_multipart`** に送信
  - **`Send as base64` ON**: ファイルは base64 エンコードされ **`POST /run`** の JSON ボディに `attachments` として同梱
- 添付なしの送信は従来通り `POST /run`
- サーバ側は WebUI と同じ経路（`in_contents` として `DigiMatsuExecute_Practice` に渡す）で処理
- レスポンスの `attachments_processed` に受理されたファイル名一覧が返り、吹き出し下のメタ欄に `attached=N` として表示

**セッションサマリの操作（左サイドバー "Session Summary" 欄）**:
- `Enable for this session` チェックを ON にすると、この Send で `session_summary_enabled: true` が `/run` の body に同梱されます
- `Preset` セレクタでテンプレートを選ぶと、その `template` も同送されて次のターンから使われます
- テンプレの中身を確認・編集したり、生成済みのサマリ内容を見るのは **6-8. Summary タブ** から

**参照情報の閲覧（References ドロワー）**:
- エージェントの応答吹き出しをクリックすると、右側に **References** ドロワーがスライドインします
- 右上の **×** ボタンで閉じます
- 参照が1件以上あった応答には `🔎` の小さなアイコンが表示され、クリック可能であることを示します
- ドロワー内は以下のセクションに分かれます:
  - **Knowledge** — RAG で参照した知識チャンク（`rag_name` / `chunk_id` / `category`）
  - **PageIndex** — PageIndex 経由で選ばれたページ（`page_id` / `summary`）
  - **Web** — Web 検索で見つけた URL 一覧（クリックで新規タブ）
  - **User Memory** — 参照したユーザーメモリ項目
- **強調表示**: 各セクション内で類似度スコア（`similarity_response`）順に並び、
  - **strong**（上位 33%）: 濃い青のバー＋薄青の背景
  - **medium**（次の 33%）: 薄い青のバー
  - **mild**（残り）: グレーのバー
- 各項目には `sim 0.92` のようにスコアが小さく表示され、`chunk_id` / `page_id` / `category` などのタグも見えるので、どのソースから来た情報かひと目でわかります
- **狭幅（<640px）**: ドロワーはチャットエリア全体を覆う画面（オーバーレイ）になります

### 6-2. Sessions — 会話履歴一覧

**やりたいこと**: `GET /sessions` でセッション一覧、行クリックで `GET /sessions/{id}` の詳細JSON

**基本操作**:
1. 上部のフィルタ（User ID / Service ID）を任意で入力（空欄=全件）
2. **GET /sessions** ボタン
3. 左側テーブルに一覧が表示される
4. 行をクリックすると右側に会話履歴の生 JSON

**プレゼン向けチップス**:
- 事前に何ターンか会話しておくと、リアルなセッションが並んで説得力が出る
- `USER_ID` フィルタで特定ペルソナに絞ると「マルチユーザ対応」を強調できる

**やりたいこと**: `GET /agents` でエージェント一覧、行クリックで `GET /agents/{file}/engines`

**基本操作**:
1. **GET /agents** ボタンで左に一覧
2. 行をクリックすると右に LLM と ImageGen のエンジン設定 JSON

**プレゼン向けチップス**:
- 「モデル切替可能」「Azure OpenAI / OpenAI / Anthropic を1つの UI から」というメッセージを見せる場面

### 6-4. Web Search Engines — Web検索エンジン設定

**やりたいこと**: `GET /web_search_engines` の返却をそのまま表示

**基本操作**: **GET /web_search_engines** ボタンを押すだけ

**プレゼン向けチップス**:
- Perplexity / OpenAI / Google 等の複数エンジン対応を示す小ネタとして

### 6-5. Feedback — 応答評価の投稿

**やりたいこと**: エージェントが持つ feedback 設定を確認し、実際に投稿する

**基本操作**:
1. **Agent file** で対象エージェントを選択
2. **GET /agents/{file}/feedback** で feedback 定義を表示
3. 右側に投稿フォーム:
   - **Session ID**: 対象の会話ID（Chat タブで自動採番されたものをコピペ）
   - **seq / sub_seq**: 対象ターン（初回応答なら `1 / 1`）
   - **feedbacks (JSON)**: 定義に沿った JSON（例: `{"rating": 5, "comment": "..."}`）
4. **Submit** で `POST /feedback`

### 6-6. Health / Raw API — 何でも屋

**やりたいこと**: 上記画面でカバーされていない API を直接叩く

**基本操作**:
- **GET /health** ボタンでヘルスチェックのみ実行
- **Raw fetch** 行で任意の method / path / body を送信可能
  - GET なら body 欄は無視
  - POST なら下の JSON テキストエリアが送信ボディ

**プレゼン向けチップス**:
- 開発者向けの説明で「拡張はここから」を示す
- 未公開エンドポイントの探索・デバッグにも便利

### 6-7. Recording Editor — 録画を画面上で編集

**やりたいこと**: 既存の録画を選び、応答文言・タイミング・イベント順序などを画面上で編集する。ファイルを外部エディタで開く必要はありません。

**画面構成**:
- 上部ツールバー: 録画のロード / 新規作成 / Apply / ダウンロード / インポート
- Meta フォーム: `id` / `title` / `createdAt` / `backendUrl`
- 左: **Events テーブル**（Δt / method / path / status / response プレビュー + 行アクション ↑↓⧉✕）
- 右: **Raw JSON エディタ**（テキストエリアで直接編集）

**基本操作**:
1. **Load recording** で編集したい録画を選び **Load into editor** をクリック
2. 左テーブルの行操作:
   - **↑ / ↓** = 順序入れ替え
   - **⧉ (Duplicate)** = 直後に複製（`t` を +500ms 自動シフト）
   - **✕ (Delete)** = そのイベントを削除
3. **+ Add event** で末尾にテンプレートイベントを追加
4. 細かい応答文言やリクエスト JSON は右側の **Raw JSON** で直接編集 → **Sync from JSON →** で左テーブルに反映
5. **Apply (register)** で in-memory の録画セレクタに反映。すぐに Play タブで再生確認できる
6. 気に入ったら **⬇ .js**（`recordings/` に配置する形式）/ **⬇ .json**（pure JSON）/ **⬇ .md**（人が読める Markdown、いずれも Import で再取り込み可）でダウンロード

**便利な使い方**:
- 一度録画した内容の**応答文言だけ差し替える**（例: 顧客名の伏せ字化）
- **タイミングを詰める**（Raw JSON で `t` を短縮）
- **エージェントの応答を空想シナリオに書き換える**（プレゼン用の完璧な回答）
- **失敗ケースを追加**（`status: 500` のイベントを Duplicate で作りエラー表示を見せる）

**注意**: **Apply する前に .js / .json ダウンロード** を忘れずに。編集内容はリロードで消えます。恒久的に組み込むには、ダウンロードした `.js` を `recordings/` に置き `index.html` に `<script src>` を追加してください。

### 6-8. Video (MP4) — 録画からアニメーション動画を書き出す

**やりたいこと**: 保存済みの録画（チャットの流れ）を、そのまま**アニメーション動画ファイル**として書き出す。LLM への再生成や再問い合わせは一切行わず、**録画済みの内容を再生してキャプチャするだけ**です。

**画面構成**:
- 上部ツールバー: `Recording`（対象録画）/ `Speed ×`（再生速度）/ `Resolution`（解像度）/ `FPS` / **Generate & Download** / Cancel
- 左: **プレビュー用 canvas** + 進捗バー + ステータス
- 右: 動作説明（現在の出力フォーマットも表示）

**基本操作**:
1. `Recording` で動画にしたい録画を選ぶ
2. `Speed ×` を設定（**下記参照**）、必要なら解像度・FPS を選ぶ
3. **● Generate & Download** をクリック
4. 録画（キャプチャ）がリアルタイムで進行 → 完了すると自動的に動画ファイルがダウンロードされる

**Speed ×（再生速度）の考え方**:
- **`1` = 人が普通に読める速さ**（基準）。吹き出しが順に現れ、エージェントの返信は「入力中…」のあと 1 文字ずつ表示されます。
- **`2`** = 2倍速（動画尺は約半分）、**`0.5`** = 0.5倍速（ゆっくり）。0.25〜16 の範囲で指定可能。
- キャプチャは実時間で行われるため、`×1` の長い会話はその長さだけキャプチャ時間がかかります（`×2` ならおよそ半分）。

**出力フォーマット**:
- 対応ブラウザ（Chrome / Edge / Safari）では **MP4** で保存されます。
- MP4 エンコーダを持たないブラウザ（Firefox など）では自動的に **WebM** にフォールバックします。現在の出力形式は右パネルに表示されます。
- ファイル名は `<録画ID>_x<速度>.<mp4|webm>` の形式。

**注意 / 制約**:
- 動画化されるのは **チャットのやり取り（`/run` イベント）** です。Sessions/Agents などの API イベントは動画には含まれません（チャットが 1 件も無い録画はエラーになります）。
- キャプチャ中は**このタブを前面に保って**ください（バックグラウンドだと描画が間引かれます）。
- 追加ライブラリ不要・完全クライアント処理なので、`file://` で開いた状態でも動作します。

### 6-8. Summary — セッションサマリの編集

**やりたいこと**: セッションごとに維持される Markdown サマリの **有効化 / テンプレート編集 / 生成内容の確認** を、`/session_summary_presets` `/sessions/{id}/summary` (GET/POST) を叩いて行う。

**画面構成**:
- 上部ツールバー: プリセット取得 / セッションID / GET でロード
- 左: 現在の状態（enabled トグル / updated_at）＋ プリセット選択 → Apply preset → で `#summary-template` に流し込み → **POST** で保存
- 右: 生成された `content`（読み取り専用プレビュー）

**基本操作**:
1. **GET /session_summary_presets** でプリセット一覧をロード（プリセットセレクタが Chat タブの Sidebar と共用される）
2. **Session ID** を入力（Chat タブで会話中の Session ID は自動でここに転記される）
3. **GET /sessions/{id}/summary** で現在の設定＋生成内容を読み込み
4. **Preset** から選んで **Apply preset →** でテンプレートを流し込み、もしくは Textarea で手編集
5. **POST /sessions/{id}/summary** で保存 → 次のターンから新テンプレで再生成される

**注意**: `content` は `DigiMatsuExecute_Practice` のバックグラウンド更新が所有します。ここで編集はできません（テンプレートを直せば次のターンから反映）。



### 7-1. なぜ録画するのか

- **オフラインで見せられる**: ネット無し／バックエンド無しでもデモが成立する
- **再現性**: 「あの時のあの応答」を毎回同じスクリプトで見せられる
- **安全性**: 実行時にトークン費用が発生しない、実データを汚さない

### 7-2. 録画の流れ

1. ヘッダー **● Record** をクリック
2. タイトルを入力（例: `顧客A向け 5分デモ`）してOK
3. モードバッジが 🔴 **REC** に変わる
4. **通常通り画面を操作** — Chat で 3～5ターン、Sessions を開く、Agents を眺める、など
5. ヘッダー **■ Stop** をクリック
6. モードバッジが 🟢 **LIVE** に戻り、録画がバッファに保存される
7. **⬇ Save** で `rec_XXXXXXXXXX.md`（Markdown）としてダウンロード

### 7-3. 録画ファイルを恒久的に組み込む

一時的に使うだけなら **⬆ Load** で `.md` を読み込めば OK（リロードで消える）。恒久的に組み込む（リロードしても常設）には、**`<script>` で読み込める `.js` 形式**が必要です:

1. Recording Editor タブで対象録画を Load し、**⬇ .js** でダウンロード（ヘッダー ⬇ Save の `.md` は `<script>` 読込不可のため）
2. ダウンロードした `.js` を `recordings/` フォルダに移動
2. `index.html` の末尾付近を編集し、既存の `sample_chat.js` の下に 1 行追加:
   ```html
   <script src="recordings/rec_XXXXXXXXXX.js"></script>
   ```
3. ブラウザをリロード
4. ヘッダーのセレクタに新しい録画が現れる

### 7-4. 録画の再生

1. ヘッダーのセレクタから録画を選択
2. **▶ Play** をクリック
3. モードバッジが 🟡 **PLAY** になり、録画通りの順序・タイミングで画面が動く
4. Chat タブに自動遷移、吹き出しが順に表示される（Sessions 系イベントが混ざっていればそのタブに切替）

**再生を途中で止める**: **■ Stop** をクリック

**速度調整**: `config.js` の `PLAYBACK_SPEED` を変更（`2.0` で2倍速、`0.5` で半分速度）

### 7-5. 録画されるもの / されないもの

**録画される**:
- API 呼び出し（method / path / request body / response body / status / エラー / タイムスタンプ）

**録画されない**:
- UI 上の入力操作そのもの（テキスト入力・ボタンクリック等）
  → 再生時は API 応答を順に「見せる」動きになります
- ネットワークの外にある副作用（バックエンドが行った DB 書き込みなど）

## 8. オフラインでのデモ運用

**ゴール**: バックエンド不在の環境で、あたかも本物が動いているように見せる。

### 8-1. 事前準備（1回だけ）

1. **開発環境で本番と同じ会話シナリオを一度実行し、録画** する
2. ダウンロードした `.js` を `recordings/` に配置、`index.html` にスクリプトタグを追加
3. `demo/sample_demo/` フォルダごと USB / メール / GitHub 等でオフラインPCへ持ち込み

### 8-2. デモ本番

1. `python3 -m http.server 8000` でローカル起動
2. ヘッダーのセレクタで **録画を選択状態にしておく**
3. `config.js` の `AUTO_FALLBACK_TO_RECORDING` が `true` であることを確認（デフォルトtrue）

これで:
- **▶ Play を押した場合**: 録画通りに画面が自動で進む（手離しデモ）
- **手動で操作した場合**: 通常通り Chat 等を操作する → 実 API はネットワーク未接続で失敗 → 自動で **録画から一致するレスポンスが返される** → あたかも動いているかのように見える

### 8-3. 注意

- 録画には response 内容がそのまま入っている ため、**実データや顧客情報を含まない** シナリオで録画すること
- 録画は「同じ method + path なら最初の一致を返す」単純マッチング。細かい引数違いは区別しない
- パスワード等の秘密情報が含まれていないか、`.js` を目視で確認してから配布する

## 10. 録画ファイルを手で編集する

> **簡単に済ませたい場合は [6-7. Recording Editor](#6-7-recording-editor--録画を画面上で編集) を使ってください**。以下は外部エディタで直接いじる場合のファイル形式解説です。

録画は普通の JS ファイルなので、テキストエディタで開いて編集できます。応答文言を直したい場合や、シナリオを一部差し替えたい場合に便利です。

### 10-1. ファイル形式

```javascript
window.Recorder && window.Recorder.register({
  id: "my_demo",
  meta: {
    title: "My Custom Demo",
    createdAt: "2026-07-01T09:00:00Z",
    backendUrl: "http://localhost:8899"
  },
  events: [
    {
      t: 200,           // 録画開始からの経過ミリ秒
      type: "api",      // 現状 "api" のみ
      method: "GET",
      path: "/health",
      status: 200,
      request: null,    // GET は基本 null
      response: { status: "ok" }
    },
    {
      t: 2000,
      type: "api",
      method: "POST",
      path: "/run",
      status: 200,
      request: { user_input: "こんにちは", agent_file: "agent_10Sample.json", /* ... */ },
      response: {
        session_id: "APIDEMO20260701...",
        session_name: "(User:DemoUser)...",
        response: "こんにちは。..."   // ← ここが吹き出しに出る文言
      }
    }
  ]
});
```

### 10-2. よくある編集

**A. 応答文言を差し替える**
`events[N].response.response` の文字列を書き換える。それだけ。

**B. 再生タイミングを詰める**
`events[N].t` の値を小さくすると、次のイベントが早く発火します。

**C. 新しいターンを追加**
`events` 配列に新しい `{t, type, method, path, request, response, status}` オブジェクトを追加。`t` は先頭からの経過ミリ秒で、時系列順に並べる。

**D. シナリオを分岐させる**
録画ファイルを `scenario_A.js` / `scenario_B.js` に分けて、それぞれ別 `id` で `register`。セレクタから選び分けられる。

## 11. 設定 (`config.json`) リファレンス

**編集対象は `config.json`**（pure JSON）です。`config.js` は起動時に config.json を同期読み込みするだけのローダーなので、通常は触りません。

```json
{
  "BACKEND_URL": "http://localhost:8899",
  "DEFAULT_USER_ID": "DemoUser",
  "DEFAULT_SERVICE_ID": "DEMO",
  "FALLBACK_AGENTS": [
    { "file": "agent_10Sample.json", "name": "Sample Agent" }
  ],
  "PLAYBACK_SPEED": 1.0,
  "AUTO_FALLBACK_TO_RECORDING": true,
  "HEALTH_POLL_MS": 0
}
```

| キー | 型 | 既定 | 説明 |
|------|-----|------|------|
| `BACKEND_URL` | string | `"http://localhost:8899"` | FastAPI のベース URL。末尾スラッシュは自動除去。ヘッダー欄で上書きすると localStorage に保存される |
| `DEFAULT_USER_ID` | string | `"DemoUser"` | Chat タブの User ID 初期値 |
| `DEFAULT_SERVICE_ID` | string | `"DEMO"` | Chat タブの Service ID 初期値 |
| `FALLBACK_AGENTS` | array | `[{file:"agent_10Sample.json",name:"Sample Agent"}]` | `/agents` の呼び出しに失敗したときに使うエージェント選択肢 |
| `PLAYBACK_SPEED` | number | `1.0` | 録画再生の速度倍率（`2.0`で2倍速、`0.5`で半分） |
| `AUTO_FALLBACK_TO_RECORDING` | boolean | `true` | 実 API 失敗時に録画から自動応答するか |
| `HEALTH_POLL_MS` | number | `0` | `>0` にすると自動的に Health チェックを繰り返す（例: `15000` で15秒毎）。`0` で無効 |

### 11-1. `file://` で開いた場合の挙動

`config.json` は `fetch()` で同期読み込みされます。`file://` プロトコルではブラウザのセキュリティ制約により取得できないケースが多いため、その場合は `config.js` に埋め込まれた **DEFAULTS** にフォールバックし、コンソールに警告が出ます。

**config.json の編集を反映したいときは必ず HTTP サーバ経由で開いてください**（`python3 -m http.server` など）。

## 12. トラブルシューティング

| 症状 | 原因 | 対処 |
|------|------|------|
| Health ドットが 🔴 | バックエンド未起動 or URL 誤り | ヘッダー欄の URL を確認、`python DigiM_API.py` で起動 |
| CORS エラー（コンソール） | 別オリジンから叩いている | FastAPI 側で CORS を許可、または同一オリジン配信 (`python -m http.server`) |
| Chat で応答が来ない | セッションが LOCKED | 少し待って再送。頻発するなら `LOCK_WAIT_MAX` 見直し |
| Send した瞬間に固まる | 実バックエンドで LLM 呼び出しに時間がかかっている | 数十秒待つ。NGINX 経由なら `proxy_read_timeout` を伸ばす |
| 録画が再生できない | セレクタ未選択、または `script src` を書き忘れ | セレクタで選択、`index.html` の script タグ確認 |
| 再生時、Sessions タブに切り替わらない | 録画に `/sessions` 系イベントが含まれていない | 録画時に該当タブを触っておく |
| リロードすると設定が消える | localStorage が無効化されている | プライベートブラウジング等を解除 |
| `⬇ Save` した .js を読み直したら空 | エクスポート前に **■ Stop** を押していない | 必ず Stop → Save の順で |
| Agents ドロップダウンが `agent_10Sample.json` しか出ない | バックエンド接続失敗 → fallback | 実バックエンドを起動、または `FALLBACK_AGENTS` を拡充 |

## 13. FAQ

**Q. デモ中に想定外の応答が返ってきたら？**
A. **F5 でリロード**、または `session_id` 欄を空にしてから再送すると新規セッションになります。事前に録画を用意しておくのが最も安全です。

**Q. 複数人が同時にデモ画面を触っても大丈夫？**
A. **画面側は完全にクライアント処理** なので競合しません。ただし実バックエンドを共有していると同じセッションを取り合うことがあります。人ごとに `USER_ID` を分けてください。

**Q. スマホやタブレットでも動く？**
A. レスポンシブは最小限です。狭幅ではタブが横スクロール、Chat のサイドバーが窮屈になります。プレゼン用途ならプロジェクター接続のPCが前提です。

**Q. 録画ファイルに機微情報が入っていないか確認したい**
A. `.js` ファイルはただのテキストです。エディタで開いて `response.response` の文字列や `request` の内容を目視確認してください。

**Q. 新しい API エンドポイントを追加したい**
A. `INSTRUCTIONS.md` の「新しいエンドポイントを画面に追加するときは」節を参照。`js/api.js` にヘルパーを増やし、`index.html` に UI を追加し、`js/app.js` にハンドラを追加、必要なら `handlePlaybackEvent` に分岐追加、の4ステップです。

**Q. デザインを大幅に変えたい**
A. `INSTRUCTIONS.md` を丸ごと Copilot / Claude に渡して指示してください。触ってよい範囲と不可侵の契約が書かれているので事故が起きにくいはずです。

**Q. 実バックエンドと録画を混ぜて使いたい（ここは実データ、ここは録画）**
A. `AUTO_FALLBACK_TO_RECORDING=true` にしておけば、実 API が成功する限り実データが返り、失敗したときのみ録画にフォールバックします。実バックエンド側で意図的に一部エンドポイントを止めれば「ここだけ録画」も実現可能です。
