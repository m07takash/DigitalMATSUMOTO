# プロジェクトマネジメントの要諦

## 概要

コンサルティングプロジェクトの成否は、優れた分析や提言だけでなく、確実なプロジェクトマネジメントによって決まる。本章では、WBS、RACI、週次レビュー、課題管理、品質管理の5つの要諦を解説する。

## WBS（Work Breakdown Structure）

WBSは、プロジェクトの成果物と作業を階層的に分解する手法である。

### 作成手順
1. 最終成果物を定義する
2. 成果物を主要なカテゴリに分解する（レベル1）
3. 各カテゴリをさらに詳細なタスクに分解する（レベル2〜3）
4. 各タスクに工数・期間・担当者を割り当てる
5. タスク間の依存関係を整理する

### 作成のコツ
- 分解の粒度は「1タスク = 1〜5営業日」が目安
- 100%ルール：上位の作業は下位の作業で100%カバーされていること
- マイルストーン（成果物の納品日、レビュー会議等）を明示する

## RACI（責任分担マトリクス）

各タスクの役割分担を明確にするフレームワーク。

| 記号 | 役割 | 説明 |
|---|---|---|
| R（Responsible） | 実行責任者 | タスクを実際に遂行する人 |
| A（Accountable） | 説明責任者 | タスクの最終承認権を持つ人（各タスクに1名） |
| C（Consulted） | 相談先 | 実行前に意見を求められる人 |
| I（Informed） | 報告先 | 実行結果を報告される人 |

**運用のポイント：**
- Aは各タスクに必ず1名のみ設定する
- RとAが同一人物でも構わないが、明示的に記載する
- プロジェクト開始時に全メンバーで合意し、変更時は速やかに更新する

## 週次レビューの運営

### アジェンダ（所要時間：60分以内）
1. **進捗確認（15分）：** 計画vs実績の差異を確認
2. **課題・リスク共有（15分）：** 新規発生した課題とリスクを共有
3. **次週の計画（15分）：** 翌週の主要タスクと担当を確認
4. **意思決定事項（15分）：** エスカレーション事項の判断

### 運営のコツ
- 会議の冒頭でゴールとアジェンダを明示する
- 事前に進捗報告を提出させ、会議では議論に集中する
- 決定事項と宿題を必ず議事録に記録し、翌営業日中に配布する

## 課題管理

### 課題管理表の必須項目
- 課題ID、起票日、起票者
- 課題の内容（具体的に記述）
- 影響度（高・中・低）、緊急度（高・中・低）
- 担当者、期限
- ステータス（未着手・対応中・完了・保留）
- 対応内容と結果

### 運用ルール
- 課題は発見次第即座に起票する（口頭報告のみは禁止）
- 週次レビューで全課題のステータスを確認する
- 期限超過の課題はエスカレーション対象とする

## 品質管理のポイント

### 成果物の品質基準
1. **論理的整合性：** 分析結果と提言が論理的に一貫しているか
2. **事実の正確性：** データや引用に誤りがないか
3. **網羅性：** 検討すべき論点が漏れなくカバーされているか
4. **実行可能性：** 提言が現実的に実行可能な内容か
5. **表現の明確性：** 読み手が誤解なく理解できる記述か

### レビュープロセス
- **セルフレビュー：** 作成者自身によるチェック（チェックリスト活用）
- **ピアレビュー：** チームメンバー間の相互レビュー
- **マネージャーレビュー：** プロジェクトマネージャーによる最終確認
- **クライアントレビュー：** 納品前のクライアント確認

## まとめ

プロジェクトマネジメントは地味だが、プロジェクトの成功を支える基盤である。これらの基本動作を愚直に実践し続けることが、信頼されるコンサルタントへの第一歩となる。
