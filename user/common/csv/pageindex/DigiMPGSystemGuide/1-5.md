# ベクトルDB管理（ChromaDB）

## 概要

ChromaDBをベクトルデータベースとして使用し、RAGナレッジの保存・検索を行う（`DigiM_Context.py`）。コサイン距離ベースの類似度検索、メタデータフィルタ、日付条件付きメタ検索、Privateモードをサポートする。

## 仕組み

### クライアント管理 (`get_chroma_client`)

`PersistentClient` をスレッドセーフなシングルトンとして管理する。ダブルチェックロッキング + `threading.Lock` で排他制御。保存先は `setting.yaml` の `RAG_FOLDER_DB`。

### 検索処理

**`_build_where_limitation(rag_data, exec_info, define_code)`**: RAGの`FILTER`定義から ChromaDB の `where` 条件を事前構築する。
- `SERVICE_INFO.ITEMS`: 各カラムを `exec_info["SERVICE_INFO"]["SERVICE_ID"]` と一致でフィルタ
- `USER_INFO.ITEMS`: 各カラムを `USER_ID` と一致でフィルタ
- `DEFINE_CODE.CODES` + `PATTERN`（`and`/`or`）: 任意キー値の組み合わせフィルタ

**`_query_collection_single(collection, query_vec, ...)`**: 1クエリベクトルに対するChromaDB検索の単位処理。
1. **メタ検索（META_SEARCH）**: `rag_data["META_SEARCH"]["CONDITION"]` の `DATE` と `meta_searches` の日付範囲を `create_date_ts` でフィルタし、距離に `rag_data["META_SEARCH"]["BONUS"]` を乗算して優先度を上げる（`query_mode="(META_SEARCH:...)"` がログに残る）
2. **通常検索**: `query_embeddings` によるベクトル類似度検索。`where_limitation` でService/User/カスタム条件を適用

**`_process_rag_data()`**: 1コレクションに対して全 `query_vecs` を ThreadPoolExecutor で並列クエリ。**Private Modeでは `{"private": {"$ne": True}}` を where に追記**して非公開データを除外。

**`create_rag_context()`**: 全RAGデータセットをコレクション単位で並列処理し、ID重複は類似度で解決後、`select_rag_vector()` でトークン上限内のチャンクを選択する。

### データ登録 (`save_rag_chunk_db`)

`get_or_create_collection(name, metadata={"hnsw:space": "cosine"})` でコレクションを作成。既存データとの差分を検出し、`title`/`key_text`/`value_text` 変更があればベクトル再計算、メタデータのみ変更なら `collection.update()` で効率更新。`embed_texts_batch()` で一括ベクトル化。`private` 未設定のレコードは自動で `False` を付与（`migrate_add_private_flag()` で一括移行も可能）。

## 設定場所

- DB保存先: `setting.yaml` の `RAG_FOLDER_DB`
- RAGマスター: `system.env` の `RAG_MST_FILE`（`rags.json`）
- RAG定義: エージェントJSON `KNOWLEDGE` / `BOOK` セクション

## 設定例

```yaml
RAG_FOLDER_DB: "user/common/rag/chromadb/"
```

```json
"category_items": [{"RAG_Category": ["memo"]}],
"private": false
```
