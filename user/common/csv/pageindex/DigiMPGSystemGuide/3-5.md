# 3-5. 会話メモリの選択と圧縮

## 概要
会話メモリは、セッション内の過去の発話をトークン予算内で選択し、LLMプロンプトに組み込む仕組みである。ロール・優先度・類似度に基づくフィルタリングと、ダイジェスト圧縮によりコンテキストウィンドウを効率的に活用する。

## 仕組み
`DigiM_Session.py`の`get_memory`メソッドが中心処理を担う。

0. **メモリ参照対象の絞り込み**: `chat_history_active_dict` から `SETTING.MEMORY_FLG == "N"` のseqを除外する。これにより、UI上の `Memory Off(seq:K)` チェックボックスでONにしたseqは画面には残るが LLM コンテキストには含まれない。
1. **ダイジェスト取得**: `memory_digest="Y"`の場合、最新のダイジェスト（会話要約）を最優先でメモリに追加する。ダイジェストのトークン数が上限を超えなければ採用される。
2. **ロールフィルタ**: `memory_role`パラメータで対象を絞る。`both`はuser/assistant両方、`user`はユーザー発話のみ、`assistant`はアシスタント応答のみを選択する。
3. **優先度モード**: `memory_priority`で選択順序を制御する。
   - `latest`: タイムスタンプ降順（最新優先）
   - `oldest`: タイムスタンプ昇順（古い順）
   - `similar`: クエリとのベクトル類似度が高い順（`memory_similarity=True`時のみ有効）
4. **トークン予算**: `memory_limit_tokens`を超えない範囲で優先度順にメモリを積み上げる。
5. **最終ソート**: 選択されたメモリをタイムスタンプ昇順に並び替えてLLMに渡す。

類似度計算には`memory_similarity_logic`（デフォルト: cosine）が使われ、各発話のベクトルファイル（.npy）をセッションフォルダから読み込む。

### ダイジェスト生成（インクリメンタル方式）

`_run_digest_background` でバックグラウンド実行されるダイジェスト生成は、毎回フル履歴を要約し直すのではなく **「前回ダイジェスト + 今回1往復」** だけをLLMに渡す方式（`DigiM_Execute.py`）。

- `session.get_history_digest(seq, sub_seq)` で **直前の最新ダイジェスト** を取得し、`{"role": "assistant", "text": ...}` として最初に積む
- そこに今回ターンの `user_query` と `assistant response` を追加し、最大3要素のスリムな入力で LLM を呼び出す
- 入力トークンが大幅に削減されるため、ダイジェスト生成時間が短縮される
- ダイジェスト生成の **実行時間** は `digest_chat_dict.timestamp_start` / `timestamp` に記録され、Detail Information の【会話のダイジェスト】に表示される

## 設定場所
エージェントJSONの`ENGINE.LLM.MEMORY`セクションで設定する。

## 設定例
```json
{
  "ENGINE": {
    "LLM": {
      "MEMORY": {
        "limit": 4000,
        "role": "both",
        "priority": "latest",
        "similarity_logic": "cosine",
        "digest": "Y"
      }
    }
  }
}
```
