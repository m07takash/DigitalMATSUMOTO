# RAGチャンク管理とPrivateフラグ

## 概要
ChromaDBへのRAGチャンク保存時に差分ベクトル化を行い、不要なAPI呼び出しを削減する仕組み。加えて、各チャンクに`private`フラグを付与してプライベートデータの管理を実現する。

## 仕組み
`save_rag_chunk_db()`は以下のロジックで動作する。

1. **既存データの取得**: ChromaDBコレクションから全IDとメタデータを`existing_map`として取得。
2. **差分判定**: 各チャンクについて以下を判定する。
   - **新規チャンク**: `chunk_id`が`existing_map`に存在しない場合 → ベクトル化必要。
   - **テキスト変更**: `title`/`key_text`/`value_text`のいずれかが既存と異なる場合（`text_changed`）→ ベクトル化必要。既存レコードを削除してから再追加する。
   - **メタデータのみ変更**: テキスト以外のフィールドが変更された場合 → `collection.update()`でメタデータのみ更新（ベクトル化なし）。
   - **変更なし**: スキップ。
3. **一括ベクトル化**: ベクトル化が必要なチャンクは`embed_texts_batch()`で`key_text`と`value_text`をまとめてAPI呼び出しし、コスト効率を最適化する。
4. **Privateフラグ**: チャンクに`private`キーがなければ`False`をデフォルト設定する。Notion側では`プライベートChk`プロパティで管理される。

### マイグレーション
`migrate_add_private_flag()`は、既存の全コレクションを走査し、`private`フィールドが未設定のチャンクに`private=False`を一括付与するマイグレーション関数。

## 設定場所
- 処理関数: `DigiM_Context.py` の `save_rag_chunk_db()`
- Privateフラグ: `rags.json`の`item_dict`内の`"private": {"プライベートChk": "chk"}`
- ChromaDBパス: `setting.yaml`の`RAG_FOLDER_DB`
- マイグレーション: `DigiM_Context.py`の`migrate_add_private_flag()`

## 設定例
```json
"item_dict": {
  "title": {"名前": "title"},
  "value_text": {"メモ": "rich_text"},
  "private": {"プライベートChk": "chk"}
}
```
