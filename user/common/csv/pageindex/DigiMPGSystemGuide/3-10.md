# ユーザーメモリ（History/Nowaday/Persona の階層的ユーザー理解）

## 概要

セッションを横断してユーザーの特徴・関心・価値観・口調を蓄積し、以降のチャットへ自動的にコンテキストとして注入する3層メモリ機構。各層は粒度・寿命・更新頻度が異なり、組み合わせて「対話相手についての情報」をプロンプト先頭に挿入する。

## 仕組み

### 3層構造

| 層 | 単位 | 内容 | 生成タイミング |
|----|------|------|---------------|
| **History** (`session_digest`) | 1セッション=1レコード | トピック / 発言抜粋 / 軸タグ（interests / values / constraints / style）/ **プルチック感情リスト(emotions)** / confidence | セッション終了時 or 手動 |
| **Nowaday** (`period_profile`) | 期間（YYYY-MM or rolling_<N>d or since_<date> or all）。**生成ごとにスナップショットを追記（履歴方式）** | 継続トピック / 新規関心 / 減退話題 / 態度の変化 / **基本8感情(強度)** / **二次感情** / 要約段落 | 月次バッチ or 手動（都度スナップショット） |
| **Persona** (`persona_profile`) | 1ユーザー=1レコード | 役割 / 専門 / 関心 / 価値観 / 制約 / 口調 / 避けたい話題（各項目に confidence と status）/ **Big5(5特性×score/confidence/status)** + summary_text | Nowaday更新時に差分マージ |

### Nowaday はスナップショット履歴方式

Nowaday は `id = service_id__user_id__period__<生成時刻>` で、**生成（Update User Memory / スケジューラ実行）のたびに新レコードを追記**する。同じ period を作り直しても過去のスナップショットは上書きされず履歴として残り、generated_at で時系列に追える。コンテキスト注入・Explorer分析・Personaマージは常に **active='Y' のうち generated_at 最新** の1件を選択する。増分生成時は同 service/user/period の最新スナップショットを既存値としてLLMに渡す。RDB は旧 `UNIQUE(service_id,user_id,period)` 制約を撤廃（初回アクセス時に自動 DROP）。

### 感情モデル（プルチックの感情の輪）と Big5

- **基本8感情**: joy(喜び)/trust(信頼)/fear(恐れ)/surprise(驚き)/sadness(悲しみ)/disgust(嫌悪)/anger(怒り)/anticipation(期待)
- **二次感情(ダイアド)**: love/submission/awe/disapproval/remorse/contempt/aggressiveness/optimism
- History.emotions=セッション単位のリスト（最大4、英語キー）。Nowaday.basic_emotions=8キー固定dict（強度0-1、History集約）、secondary_emotions=発生中の二次感情。
- **Big5**(Persona): openness/conscientiousness/extraversion/agreeableness/neuroticism を `{score:0-1, confidence:0-1, status}` で保持。リスト項目と同じ pending→approved 自動昇格規則。コンテキストには approved + pending を含める（pendingは「(暫定)」付き、deletedのみ除外）。
- History検索ではクエリ文中の感情語（日本語キーワード）も検出してマッチに加算。
- スキーマ拡張前の既存レコードは `python3 DigiM_GeneUserMemory.py --backfill` で感情/Big5を後追い投入可能（Notion時はDBプロパティ自動追加）。

### ストレージ抽象化（`DigiM_UserMemory.py`）

層ごとに独立して保存先を **Excel / Notion / RDB** から選択可能（system.envの `USER_MEMORY_<LAYER>_BACKEND`）。

- **Excel**: `user/common/user_memory/<layer>.xlsx`
- **Notion**: `notion_db.json` の `DigiM_UserMemory_History/Nowaday/Persona` キーで指定したDB（必須プロパティ: `title`(title)、その他はrich_text/number/date/checkbox任意。`active`はcheckbox、`create_date`/`generated_at`はdate型推奨）
- **RDB**: PostgreSQL の `digim_user_memory_history/nowaday/persona` テーブル（初回アクセスで自動作成）

### コンテキスト注入（`DigiM_UserMemoryBuilder.py`）

`DigiM_Execute.py` で **Knowledgeコンテキストの直前**にプロンプト先頭として挿入。構造例:

```
# 対話相手について
応答時の口調・関心・避けたい話題の参考にしてください。

## 人物像             ← Persona(approved + pending「(暫定)」付き / deletedのみ除外、各フィールドUSER_MEMORY_PERSONA_MAX_CHARS_PER_FIELD字以内)
## 最近の傾向（period） ← Nowaday
## 直近セッション      ← History(タグ×時間ハイブリッド検索、USER_MEMORY_HISTORY_MAX_CHARS字以内)
```

IMAGEGEN（画像生成）はプロンプト3000字制限を圧迫しないようスキップ。

### Historyメモリの選定（タグ×時間ハイブリッド）

埋め込みベクトルを使わず、以下のロジックで関連Historyを選定する:

1. MeCabでユーザー入力から名詞を抽出（タグ側は意味単位として保持、分解しない）
2. 各Historyの `axis_tags` 全文に対しクエリ名詞の部分一致で「マッチタグ」を判定
3. **マッチ群**: マッチ率 × (1 - α) + 時間スコア × α（α=`USER_MEMORY_HISTORY_RECENCY_WEIGHT`）でランク
4. **非マッチ群**: 時間スコアのみ（指数減衰、半減期=`USER_MEMORY_HISTORY_RECENCY_HALF_LIFE_DAYS`）
5. マッチ群を優先 → 余りを非マッチ群で埋める → 合計 `USER_MEMORY_HISTORY_MAX_CHARS` 字まで詰める

### Personaのステータスと自動承認

各項目（label/confidence/status/evidence）は3つのステータスを持つ:

| status | 意味 | コンテキスト含有 |
|---|---|---|
| **approved** | 信頼できる（ユーザー承認済 or `confidence ≥ USER_MEMORY_AUTO_APPROVE_THRESHOLD`で自動承認） | ✓ |
| **pending** | 未レビュー | ✓（「(暫定)」付きで低信頼を明示） |
| **deleted** | 不要（再提案も弾く） | ✗ |

差分マージ規則:
- approvedはラベルが保護され、信頼度のみmax更新
- pendingは新confidenceで上書き
- deletedは内部に保持され、LLMが同じラベルで再提案しても自動的に弾かれる
- 各フィールドは `USER_MEMORY_PERSONA_MAX_CHARS_PER_FIELD`（label合計）で打ち切り、approved優先

### Historyメモリの生成入力

LLMでのHistory抽出時、以下を加味する:
- セッション会話履歴（user/assistant）
- **Feedback**（role=feedback として末尾付与）— ユーザー本人意思が最も強い情報として重視
- **除外**: SETTING.FLG="N"（削除）/ MEMORY_FLG="N"（メモリ参照対象外）seq、setting.memory_flg="N" sub_seq

### 統合パイプラインとスケジューラ

- `DigiM_GeneUserMemory.update_user_memory_pipeline()` で History → Nowaday → Persona を順次実行
- スケジュールは `DigiM_Scheduler.py`（APScheduler）+ `DigiM_ScheduledJobs.py` が `user/common/mst/scheduled_jobs.json` を読み実行。`kind="user_memory_nowaday"` ジョブで Nowaday→Persona を定期実行（cron書式、再起動なしReload可）。WebUIの **Scheduler** メニューで管理（旧 `USER_MEMORY_NOWADAY_SCHEDULE` env は廃止）
- 手動: WebUI > RAG Management > User Memory の **Update User Memory** ボタン1回で全層更新。CLI `python3 DigiM_GeneUserMemory.py --backfill` で既存レコードへの感情/Big5バックフィルも可能
- 分析: **User Memory Explorer** メニュー（横断/深掘り + メモリ接地対話）。詳細は 5-5 参照

### On/Off の2階層

優先順は **ユーザーマスタ > システムデフォルト**:

| 階層 | 格納先 | 内容 |
|------|--------|------|
| システムデフォルト | system.env `USER_MEMORY_DEFAULT_LAYERS` | 全ユーザー初期値（空文字 `""` で全Off） |
| ユーザーマスタ | `users.json` の `Allowed["User Memory Layers"]` | このユーザーの有効化する層リスト |

`Allowed["User Memory"]`（true/false）でWebUI表示・編集権限をフラグ管理。

## 設定場所

- 主モジュール: `DigiM_UserMemory.py`（ストレージ）/ `DigiM_UserMemoryBuilder.py`（コンテキスト合成）/ `DigiM_GeneUserMemory.py`（生成・バックフィルCLI）/ `DigiM_UserMemorySetting.py`（設定解決）/ `DigiM_Scheduler.py`+`DigiM_ScheduledJobs.py`（スケジュール）/ `DigiM_UserMemoryExplorer.py`（分析・対話）
- LLMエージェント: `agent_62UserMemoryHistory.json` / `agent_61UserMemoryNowaday.json` / `agent_60UserMemoryPersona.json`
- プラクティス: `practice_13UserMemoryHistory.json` / `_14UserMemoryNowaday.json` / `_15UserMemoryPersona.json`
- プロンプト: `prompt_templates.json` の `User Memory History` / `Nowaday` / `Persona`
- WebUI: メイン画面 BOOK 直下の **User Memory** expander（層On/Off + Save Layer Setting のみ）、メモリ内容の確認・修正は **User Memory Explorer >「マイメモリ」タブ**（5-5）、サイドバー > RAG Management > **User Memory** 管理者ブロック（Target User IDs / Period / Update）

## 設定例

```env
# 各層のバックエンド
USER_MEMORY_HISTORY_BACKEND="NOTION"
USER_MEMORY_NOWADAY_BACKEND="NOTION"
USER_MEMORY_PERSONA_BACKEND="NOTION"
USER_MEMORY_DEFAULT_LAYERS="persona,nowaday,history"

# 自動承認 / 各種上限
USER_MEMORY_AUTO_APPROVE_THRESHOLD=0.8
USER_MEMORY_PERSONA_TOKEN_LIMIT=3000
USER_MEMORY_PERSONA_MAX_CHARS_PER_FIELD=300
USER_MEMORY_HISTORY_MAX_CHARS=800
USER_MEMORY_HISTORY_RECENCY_WEIGHT=0.3
USER_MEMORY_HISTORY_RECENCY_HALF_LIFE_DAYS=30
```

スケジュールは env ではなく `user/common/mst/scheduled_jobs.json`（WebUI Scheduler メニュー）で設定:

```json
[{"job_id": "umnowaday", "kind": "user_memory_nowaday", "cron": "0 3 1 * *", "enabled": true}]
```

users.json:

```json
"USER0001": {
  "Allowed": {
    "User Memory": true,
    "User Memory Layers": ["persona", "nowaday", "history"]
  }
}
```

## 効果ログ

LLM呼び出し時、注入元の層IDが `response.reference.user_memory` に、注入コンテキスト全文が `prompt.user_memory_context` に保存される。WebUIの **Detail Information** 内 `【ユーザーメモリ】` セクションで参照ID + 全文を確認可能。
