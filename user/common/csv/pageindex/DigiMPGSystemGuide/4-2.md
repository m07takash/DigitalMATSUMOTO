# 4-2. ストリーミング応答とバックグラウンド実行

## 概要
LLMの応答をリアルタイムに表示するストリーミングモードと、Web UIでの非同期バックグラウンド実行の仕組みを提供する。セッションロックにより同時実行を防止する。

## 仕組み

### ストリーミング応答
`DigiM_FoundationModel.py`の各LLM関数（`generate_response_T_gpt`、`generate_response_T_gemini`、`generate_response_T_claude`等）は`stream_mode`パラメータを受け取る。

- `stream_mode=True`: APIのストリーミング機能を利用し、チャンクごとに`yield`でレスポンスを返す。GPTは`stream=True`、Geminiは`generate_content_stream`、Claudeは`messages.stream`を使用する。
- `stream_mode=False`: 一括でレスポンスを取得して返す。
- Geminiではストリーミング結果が空の場合、自動的に非ストリーミングモードで再実行するフォールバック機能がある。

`DigiM_Execute.py`では2秒間隔（`_STREAM_FLUSH_INTERVAL = 2`）で中間レスポンスをセッションステータスに書き出す疑似ストリーミング機能も実装されている。

### バックグラウンド実行
`WebDigiMatsuAgent.py`で実装されるWeb UI固有の仕組み。

1. **事前ロック**: ユーザー入力時にセッションを`LOCKED`状態にし、`_PRE_LOCKED=True`を設定
2. **_run_bg関数**: `threading.Thread`でバックグラウンドスレッドを起動し、`DigiMatsuExecute_Practice`を実行。チャンクは消費するが画面には直接表示しない（結果はchat_memory.jsonに保存）。
3. **_lock_monitor**: Streamlitの`@st.fragment(run_every=2)`デコレータで2秒間隔のポーリングを実現。セッションステータスを監視し、`LOCKED`解除を検知するとフルリロードする。中間レスポンスがあればリアルタイム表示する。
4. **セッション名自動生成**: バックグラウンド完了後、セッション名が未設定なら`gene_session_name`で自動生成する。

### ジョブレジストリ（`DigiM_JobRegistry`）
RAG更新・知識活用分析・エージェント比較・会話ダイジェスト生成などのバックグラウンドスレッドを`register_job`/`unregister_job`で追跡。

- サイドバー **Sessions → Background Jobs** から稼働中ジョブの一覧表示・選択キャンセルが可能
- 一般ユーザーは自分の`user_id`のジョブのみ、Adminは全ジョブを閲覧
- キャンセルは`cancel_requested`フラグ + `PyThreadState_SetAsyncExc`による`SystemExit`注入。LLMストリームは秒単位で停止するが、C拡張のI/O中はsyscallが返るまで例外発火しない
- 別系統として、cron駆動の汎用ジョブは `DigiM_Scheduler`+`DigiM_ScheduledJobs`（WebUI上部の **Scheduler** メニュー）で管理する（詳細は 4-7）

### マルチペルソナ並列実行（`DigiMatsuExecute_MultiPersona`）
ORG/Persona選択で2人以上のペルソナを指定した場合、`ThreadPoolExecutor(max_workers=MAX_PARALLEL_PERSONAS)` で1往復をペルソナ並列で実行する:
- 事前にseqを確定（`_SEQ_OVERRIDE`）して並列ワーカー間のレース回避
- 各ペルソナはサブseq=1, 2, ..., N に保存され、`setting.persona_id`/`persona_name`で識別
- 並列パス内では `SAVE_DIGEST=False`（ダイジェスト生成スキップ）、`_NO_UNLOCK=True`（内側Practiceの個別UNLOCKを抑制）
- 完了後、seqの `MEMORY_FLG="N"` を自動セット → 次ターンの会話メモリには含まれない（画面表示は残る）
- WebUIの **Include Query** チェックボックス（直前seqが複数ペルソナ実行と判定された場合のみ表示）ONで、次ターンの `user_input` 先頭に各ペルソナ全文応答を埋め込む（RAGクエリ生成には影響なし）

### chain.PERSONAS（Phase 6/7: ステップ単位の並列）
Practice内の特定ステップにも `chain.PERSONAS` / `chain.PERSONA_MERGE` / `chain.PERSONA_MERGE_LEVEL` を指定でき、ステップ単位で複数ペルソナ並列 → マージ → 次ステップへ繋げられる:
- `chain.PERSONAS = "WEB_UI"`：UIで選択中のペルソナを使用 / `["P0001",...]` で固定指定 / `"THINKING"` で `agent_54PersonaSelector.json` がORG配下から自動選定（Phase 7、上限は `setting.yaml` の `MAX_PERSONAS`、既定3）
- `chain.PERSONA_MERGE = "summary" | "concat" | "first" | "include_query" | "none"`：並列応答を次ステップ入力にどう統合するか
- 統合用ヘルパー `DigiM_Tool.dialog_persona_merge()` と専用サポート `agent_50PersonaMerge.json`（プロンプト「Persona Merge」の `{summary_level}` プレースホルダで要約強度を制御）
- マジックワード起動の代表サンプル：`practice_51PersonaCompare.json`（**皆の意見を聞いて**）/ `practice_52PersonaThinking.json`（**適任に聞いて**）

詳細は **3-9** を参照。

### UNLOCK制御
プラクティスの最終チェインでダイジェスト生成が完了した時点でUNLOCKされる。中間チェインでは`_UNLOCK_ON_DIGEST=False`により早期UNLOCKを防止する。

## 設定場所
- `execution["STREAM_MODE"]`: ストリーミングのON/OFF
- Web UIのサイドバー設定

## 設定例
```python
execution = {
    "STREAM_MODE": True,
    "SAVE_DIGEST": True,
}
```
