# 3-9. ORG・Persona（複数ペルソナの管理と並列実行）

## 概要

1つのテンプレートエージェント（`agent_*.json`）に対して、ExcelまたはPostgreSQLで複数の **ペルソナ** を登録し、`ORG`（組織dict）の部分一致で動的に切り替える仕組み。WebUI／プラクティスから複数ペルソナを並列実行し、応答を比較・統合できる。

## 仕組み

### ペルソナマスタ（`DigiM_AgentPersona.py`）

ペルソナは `persona_id` 単位の dict で、以下の主要フィールドを持つ。

| フィールド | 型 | 用途 |
|------------|----|------|
| `persona_id` | str | 一意ID |
| `template_agent` | str | 紐付けるテンプレ（空なら全テンプレ共通） |
| `org` | dict | 組織情報（`{"company":"...", "dept":"..."}` など、自由スキーマ） |
| `company` / `dept` / `name` / `act` | str | 表示・ログ用 |
| `personality` | dict | テンプレの `PERSONALITY` を全置換 |
| `habits` | list | `["ALL"]` または `["DEFAULT","FRIENDLY"]` |
| `knowledge` | list | `["ALL"]` または `["Quote","Memo"]`（`RAG_NAME` でフィルタ） |
| `define_code` | dict | RAGフィルタ用の定義コード |
| `character_text` / `character_file` | str | 人格詳細（テキスト直記述／`character/` 配下のファイル参照） |
| `active` | `Y`/`N` | 論理削除 |

### ストレージバックエンド

環境変数 `AGENT_PERSONA_SOURCE`（既定 `EXCEL`）で切替。

| 値 | 参照先 |
|----|--------|
| `EXCEL` | `user/common/agent/persona_data/` 配下の `.xlsx`（シート名 `personas`） |
| `RDB` | PostgreSQL の `digim_agent_personas` テーブル（初回アクセス時に `CREATE TABLE IF NOT EXISTS`） |
| `BOTH` | Excel + RDB をマージ（同 `persona_id` は **RDB優先**） |

`psycopg2` は遅延importなので、`EXCEL` モード時はDB依存なし。

#### per-agent override（`PERSONA_SOURCE`）

エージェントJSONに `"PERSONA_SOURCE": "EXCEL" | "RDB" | "BOTH"` を書くと、**そのテンプレートだけ**ストレージを上書きできる。省略時は環境変数 `AGENT_PERSONA_SOURCE` にフォールバック。RDB運用環境で特定テンプレートだけExcelを使う、といった運用に対応する。

### ORGマッチング（`find_personas_by_org`）

- エージェントJSON の `ORG`（list of dict）から **WebUI/プラクティスで1つを選択**
- 選択した dict の **全キー** が `persona.org` に同値で含まれるペルソナをマッチ（agent側が persona側の **サブセット**）
- 例: persona `{company:"デジMラボ", dept:"Consulting", BU:"DX"}` は agent選択 `{company:"デジMラボ", BU:"DX"}` にマッチ

### Persona上書きの対象（テンプレートに対して）

| 上書きされる | 上書きされない |
|--------------|----------------|
| `NAME` / `ACT` / `PERSONALITY`（`character_text` または `character_file` 含む）/ `HABIT`（`["ALL"]` 以外なら名前ホワイトリストでフィルタ）/ `KNOWLEDGE`（`["ALL"]` 以外なら `RAG_NAME` でフィルタ）/ `DEFINE_CODE` | `ENGINE` / `SUPPORT_AGENT` / `BOOK` / `SKILL` / `COMMUNICATION` |

## マルチペルソナ並列実行

### 実行モード

| 起動経路 | 実装 | 用途 |
|---------|------|------|
| WebUIマルチペルソナ送信 | `DigiMatsuExecute_MultiPersona`（`DigiM_Execute.py`） | 1往復をペルソナ並列で実行（同seq内の sub_seq に保存） |
| Practice内 step並列（Phase 6） | `chain.PERSONAS` + `PERSONA_MERGE` | ステップ単位で並列→マージ→次ステップへ |
| Thinking自動選定（Phase 7） | `chain.PERSONAS = "THINKING"` + `agent_54PersonaSelector.json` | 質問内容に合わせて適任ペルソナを最大N人自動選定 |

### `chain.PERSONAS` の値と挙動

| 値 | 動作 |
|----|------|
| `"WEB_UI"` | WebUIで現在選択中のペルソナを使用 |
| `["P0001","P0002",...]` | persona_id を直接指定 |
| `"THINKING"` | PersonaSelector が **選択中ORG** 配下から最適ペルソナを最大 `MAX_PERSONAS` 人選定 |

### `PERSONA_MERGE` 戦略

並列実行されたペルソナ応答を、次ステップに渡すテキストへどう統合するか。

| 値 | 動作 |
|----|------|
| `summary` | `agent_50PersonaMerge.json` で要約。`PERSONA_MERGE_LEVEL`（`light`/`medium`/`heavy` または自由記述）で要約強度を制御。プロンプト「Persona Merge」内の `{summary_level}` プレースホルダに展開 |
| `concat` | 全ペルソナ応答を連結 |
| `first` | 最初のペルソナ応答のみを採用 |
| `include_query` | `[前回の各ペルソナの回答]\n- name: text\n... [今回の質問] ...` 形式で次ステップ入力に |
| `none` | マージ処理をスキップ |

### 並列実行・メモリ制御

- `ThreadPoolExecutor` の上限は `setting.yaml` の `MAX_PARALLEL_PERSONAS`（既定 4）
- 事前にseqを確定（`_SEQ_OVERRIDE`）して並列ワーカー間のレースを回避
- 各ペルソナは sub_seq=1, 2, ..., N に保存。`setting.persona_id` / `persona_name` / `chain_role="persona"` で識別
- 並列パス内では `SAVE_DIGEST=False`（ダイジェスト生成スキップ）、`_NO_UNLOCK=True`（内側Practice の個別UNLOCKを抑制）
- 並列sub_seq には自動的に `setting.memory_flg="N"` が付与され、**次ターンの会話メモリから除外** される（画面表示は残る）
- 次ステップは `OUTPUT_<開始sub_seq>` でマージ結果を参照

### Include Query（次ターン入力への埋め込み）

WebUIサイドバー下部の **Include Query** チェックボックス（**直前seqが複数ペルソナ実行だった場合のみ表示**）をONにすると、次ターンの `user_input` 先頭に各ペルソナ全文応答が埋め込まれる。RAGクエリ生成には影響しない。

## Phase 7: Thinking連動の自動選定

`chain.PERSONAS = "THINKING"` のとき、`agent_54PersonaSelector.json` がユーザー質問に最適なペルソナを選定する。

- **候補プール**: WebUIで選択中の **ORG** に合致するペルソナ全件
- **上限N**: WebUIサイドバーの **Max Personas**（Thinking Mode ON かつ Thinking Targets に `Personas` 選択時のみ表示。既定は `setting.yaml` の `MAX_PERSONAS=3`）
- **選定ロジック**: 質問内容と互いに補完的な視点を持つペルソナを選ぶ。1人で十分なら1人、専門外は除外
- **フォールバック**: 選定失敗 / 0人選定 → UI multiselect の選択にフォールバック
- **無効化**: WebUI Thinking Targets から `Personas` を外すと、PersonaSelector はスキップされ `chain.PERSONAS="THINKING"` ステップは `in_personas` にフォールバック
- **Detail Information**: 選定理由は `_THINKING_RESULT.personas_reason` に記録

## サンプルPractice

| ファイル | マジックワード | 動作 |
|---------|----------------|------|
| `practice_51PersonaCompare.json` | 皆の意見を聞いて | chain[0]で WebUI選択ペルソナ並列回答 → chain[1]で比較評価 |
| `practice_52PersonaThinking.json` | 適任に聞いて | chain[0]で Thinkingが自動選定したペルソナで並列回答 → chain[1]で比較評価 |

## 設定場所

- グローバル: `system.env` の `AGENT_PERSONA_SOURCE`
- per-agent: エージェントJSON の `PERSONA_SOURCE` / `ORG` / `PERSONA_FILES`
- ペルソナ実体: `user/common/agent/persona_data/*.xlsx`（EXCEL）または `digim_agent_personas` テーブル（RDB）
- 並列上限: `setting.yaml` の `MAX_PARALLEL_PERSONAS`
- Thinking選定上限: `setting.yaml` の `MAX_PERSONAS`（WebUI Max Personas入力の既定値）

## 設定例

エージェントJSON:
```json
{
  "ORG": [
    {"company": "デジMラボ"},
    {"company": "デジMラボ", "dept": "Consulting"},
    {"company": "デジMラボ", "BU": "DX"}
  ],
  "PERSONA_FILES": ["agent_X0Sample_personas.xlsx"],
  "PERSONA_SOURCE": "RDB"
}
```

Practice（Phase 6 step並列＋Phase 7自動選定）:
```json
"CHAINS": [
  {
    "TYPE": "LLM",
    "PERSONAS": "THINKING",
    "PERSONA_MERGE": "include_query",
    "PERSONA_MERGE_LEVEL": "medium",
    "SETTING": { "AGENT_FILE": "USER", "PROMPT_TEMPLATE": "Normal Template", "MEMORY_USE": true }
  },
  {
    "TYPE": "LLM",
    "SETTING": {
      "AGENT_FILE": "USER",
      "USER_INPUT": ["OUTPUT_1", "\n\n[追加指示]\n各ペルソナの主張を比較評価してください。"],
      "PROMPT_TEMPLATE": "No Template",
      "MEMORY_USE": false
    }
  }
]
```
