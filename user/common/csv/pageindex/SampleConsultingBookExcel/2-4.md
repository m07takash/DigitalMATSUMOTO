# ビジネスプロセス分析（As-Is / To-Be）

業務の流れ（プロセス）を可視化し、ボトルネック・非効率・属人化を特定して再設計するアプローチ。BPR・DX推進の基本ツール。

## As-Is → To-Be

| フェーズ | 目的 | アウトプット |
|---------|------|--------------|
| **As-Is分析** | 現状の正確な可視化 | プロセス図、所要時間、担当者、課題リスト |
| **To-Be設計** | あるべき姿の構築 | 再設計プロセス図、KPI、必要システム要件 |
| **ギャップ分析** | 移行に必要な打ち手の整理 | 施策一覧、優先順位、ロードマップ |

## 表記法

- **BPMN**（Business Process Model and Notation）：業務フローの国際標準。スイムレーン（担当者別レーン）でロール分担を明示
- **DFD**（Data Flow Diagram）：情報の流れに着目
- **業務フロー図**（簡易）：シンボル少なめで関係者と握りやすい

## 分析の観点（ECRS原則）

業務改善の4原則：
1. **Eliminate（排除）**：そもそもやめられないか
2. **Combine（結合）**：他の業務と一緒にできないか
3. **Rearrange（再配置）**：順序や担当を入れ替えられないか
4. **Simplify（簡素化）**：もっと簡単にできないか

「自動化」より先に「排除」「結合」を検討するのが鉄則。

## ボトルネックの見つけ方

- **リードタイム**：プロセス全体の所要時間
- **タッチタイム**：実作業時間
- **待ち時間** = リードタイム − タッチタイム

待ち時間の比率が大きい工程が最優先の改善対象。

## よくある失敗

- As-Isを書き込みすぎて時間切れ。**例外処理は2割まで** に留めて主要フローに集中する
- いきなりTo-Beに飛んでしまい、現場の納得感を得られない
- システム化を前提にプロセスを設計し、「業務改善」ではなく「現状システム化」になる
- ECRSの「E」を飛ばして自動化ありきで進める
